# 原神虚拟恋人平台 - 项目设想文档

## 一、需求概述

基于原神角色信息构建虚拟恋人平台，用户可与喜爱的角色进行对话互动。项目分网页版与手机版，优先完成网页版。

**核心数据来源**：原神官网/维基等公开站点
- 角色属性、故事背景、技能设定
- 角色立绘、武器、圣遗物等美术资源

**核心功能**：将角色数字化为可对话的虚拟人物，支持文字聊天、对话历史保存，后期可扩展实时语音通话。

---

## 二、用户视角

### 2.1 核心体验

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 角色选择 | 浏览并选择要互动的原神角色 | P0 |
| 文字对话 | 与角色进行自然语言对话，获得符合人设的回复 | P0 |
| 对话历史 | 查看、回溯与某角色的全部聊天记录，类似微信 | P0 |
| 角色展示 | 展示角色立绘、背景故事、技能等，增强沉浸感 | P0 |

### 2.2 期望体验

- **人设一致**：回复风格、语气、用词符合角色设定（如胡桃活泼、钟离沉稳）
- **记忆感**：能记住之前的对话内容，形成连贯关系
- **情感反馈**：根据互动频率、内容，角色态度有微妙变化（可选）
- **界面美观**：贴合原神美术风格，有代入感

### 2.3 可扩展功能（用户侧）

| 功能 | 描述 | 难度 |
|------|------|------|
| 好感度/亲密度 | 根据互动增加数值，解锁更多对话或剧情 | 中 |
| 每日签到/任务 | 每日与角色互动获得奖励，提升粘性 | 中 |
| 角色卡片/图鉴 | 收集已解锁角色，展示成就 | 低 |
| 剧情分支 | 预设剧情线，用户选择影响走向 | 高 |
| 语音消息 | 角色用合成语音回复（需语音资源） | 高 |
| 实时语音通话 | 与角色语音对话（需语音克隆/训练） | 很高 |
| 虚拟形象/立绘表情 | 根据对话内容切换立绘表情、动作 | 中 |
| 礼物/约会系统 | 送礼物、模拟约会场景 | 中 |
| 多角色关系 | 多个角色之间有关联，用户可切换互动 | 中 |

## 三、管理员视角

| 功能 | 描述 |
|------|------|

| 敏感词/合规 | 配置敏感词过滤、内容合规策略 |

## 四、技术实现

### 4.1 AI 接入方式（无需单独“训练”）

**推荐做法**：使用大语言模型（LLM）的 **System Prompt（系统提示词）** 塑造角色，而非训练专用模型。

| 方式 | 说明 | 成本 | 效果 |
|------|------|------|------|
| System Prompt | 在每次请求中注入角色人设、背景、说话风格 | 低 | 足够好，易迭代 |
| Fine-tuning | 用角色对话数据微调模型 | 高 | 更贴合，但维护难 |
| 专用数字人模型 | 自建或采购数字人方案 | 很高 | 适合语音、形象一体化 |

**结论**：先用 System Prompt + 通用 LLM（如 GPT、Claude、国产大模型）即可实现文字对话，成本低、迭代快。语音、数字人可视作后期扩展。

### 4.2 技术栈建议

| 模块 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | React / Vue + 响应式布局 | 网页版优先，便于后续做移动端适配 |
| 后端 | Node.js / Python (FastAPI) | 处理业务逻辑、会话管理 |
| 数据库 | PostgreSQL / MySQL | 用户、角色、对话历史 |
| 缓存 | Redis | 会话状态、限流 |
| AI 接口 | OpenAI / Claude / 国产大模型 API | 按地区、成本选择 |
| 存储 | 对象存储（OSS/S3） | 角色图片、用户头像等 |
| 部署 | 云服务器 / Vercel / 自建 | 根据预算选择 |

### 4.3 核心数据流

```
用户输入 → 后端 → 拼接 System Prompt + 历史对话 + 当前输入 → 调用 LLM API
         → 解析回复 → 存入数据库 → 返回前端展示
```

### 4.4 对话历史存储设计

- **表结构**：`users`、`characters`、`conversations`、`messages`
- **messages**：user_id, character_id, role (user/assistant), content, created_at
- 按用户+角色维度查询历史，限制最近 N 轮参与上下文（如 20 轮），避免 token 超限

## 五、能做到的程度

### 5.1 网页版 MVP（最小可行产品）

| 能力 | 可实现 |
|------|--------|

| 多角色切换 | ✅ 每个角色独立对话线程 |

### 5.2 中期可扩展

| 能力 | 可实现 |
|------|--------|
| 好感度、签到 | ✅ 规则与数值设计 |

## 六、开发阶段建议

1. **Phase 1**：数据采集（角色信息、图片）→ 已有 genshin_spider 可复用
2. **Phase 2**：网页前端 + 后端 API + 用户系统 + 单角色对话
3. **Phase 3**：多角色、对话历史、人设优化
4. **Phase 4**：管理后台、好感度、扩展玩法
5. **Phase 5**：语音、移动端（按需）

---

## 七、风险与注意事项

- **版权**：原神 IP 归属米哈游，商用需授权；个人/学习项目需控制传播范围
- **内容合规**：虚拟恋人场景需注意内容边界，建议接入审核
- **成本**：LLM 按 token 计费，需做限流、缓存、模型选择以控制成本
- **地区限制**：部分 AI 服务有地区限制，可参考 [Cursor Regions](https://cursor.com/docs/account/regions) 或使用国产大模型
