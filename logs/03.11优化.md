*   **流式输出 (Streaming / SSE)：** 如果目前是等待整个 LLM 响应生成完再返回结果，用户会感觉卡顿。使用 **Server-Sent Events (SSE)** 或 WebSockets 将 LLM 的回答逐字/逐句推送到前端，显著降低首字节响应时间 (TTFB)。
*   **语音交互 (TTS / ASR)：** 
    *   语音合成 (TTS)：加入 **VITS、Edge-TTS 或 CosyVoice** 等拟真语音模型，让 AI 恋人可以用角色的专属声线回复（尤其是原神角色）。
    *   语音识别 (ASR)：引入 **Whisper**，支持用户语音输入。
*   **情感分析与多模态反馈：** 对 AI 返回的文本进行情感判定（如开心、生气、傲娇），前端动态切换角色的立绘表情（Live2D / Spine 面部差分），甚至结合触觉反馈（移动端震动）。

### 2. 后端与系统性能优化 (Performance)
*   **缓存层 (Redis)：** 使用 **Redis** 缓存高频访问的只读数据（如角色面板信息、初始提示词）。同时用于优化 `rate_limit.py`（分布式限流）和保存短期 Session 状态。
*   **全双工通信 (WebSockets)：** 在 `chat.py` 中，如果目前基于普通 HTTP 轮询，建议改写为 **WebSocket**。WS 能保持长连接，极大降低聊天过程中的网络开销和延迟。
*   **异步并发架构：** 确保 FastAPI/Python 后端中的数据库操作全异步化（如使用 `asyncpg` 或 `Motor`），防止阻塞线程池。对耗时任务（如生成图片、定期总结记忆）引入 **Celery 或 RQ** 配合 RabbitMQ 进行后台异步处理。

### 3. 前端体验与渲染优化 (Frontend UX)
*   **长列表虚拟化 (Virtual List)：** 聊天记录变多时，渲染大量 DOM 节点会导致页面卡顿。在 `ChatPage.tsx` 使用 `react-window` 或 `react-virtuoso` 实现虚拟列表，仅渲染可视区域内的消息。
*   **渐进式 Web 应用 (PWA)：** 配置 `vite-plugin-pwa`，包含 Service Worker。这能让用户将“AI恋人”安装到手机主屏幕上（类似原生APP），并且可以进行离线资源缓存，加快二次加载速度。
*   **骨架屏 (Skeleton) 与过渡动画：** 在等待接口响应和图片加载阶段添加柔和的过渡动画或骨架屏，减轻用户的等待焦虑。
*   **图片进一步优化：** 你已经在 scripts 下有了图片优化脚本。前端可以进一步使用原生 `<picture>` 标签支持 **WebP / AVIF** 格式，并配合懒加载 (Lazy loading) 减少首次白屏时间。

### 4. 监控与工程化 (DevOps)
*   **实时性能与异常监控：** 引入 **Sentry**，及时捕获前端崩溃或后端报错，以免 AI 恋人“突然失联”又不知原因。
*   **用户埋点分析：** 接入轻量级的数据分析工具（如 Umami 或 Google Analytics），了解用户最喜欢的角色、平均聊天时长和高频使用时间端，为后续迭代提供依据。

### 第一阶段：实现流式输出（打字机效果）

**1. 修改后端 LLM 服务 (llm.py)**
*   当前使用的是 `dashscope` 并且是阻塞等待完整结果。
*   **改法：** 将调用改为流式： `stream=True, incremental_output=True`，并使用 Python 的 `yield` 生成器将字块逐个返回。

**2. 修改后端 API 接口 (chat.py)**
*   目前的 `/chat` 接口返回的是一个完整的 JSON `{"reply": reply}`，且在返回前就写入了数据库。
*   **改法：** 引入 FastAPI 的 `StreamingResponse`。将接口改为 SSE（Server-Sent Events）格式推送数据。在流传输完全结束后，在后台任务（`BackgroundTasks`）中把完整的对话存入数据库。

**3. 修改前端 API 客户端 (client.ts)**
*   现在的 `api.chat` 是普通的 `await fetch().then(res => res.json())`。
*   **改法：** 获取 `response.body.getReader()`，通过 `while (true)` 循环不断读取流中的二进制数据，并使用 `TextDecoder` 解析出新增的文字。

**4. 优化前端聊天页面 (ChatPage.tsx)**
*   目前是等待 `loading` 结束后一次性追加对话。
*   **改法：** 发送消息后，立刻在消息列表中插入一条空的 assistant 消息，然后根据接收到的流数据，持续更新这条最后的消息，实现视觉上的“打字机”效果。

### 第二阶段（后续）：引入长期记忆（RAG）

由于长上下文会导致 Token 消耗很大且响应变慢，我们可以在后面引入长期记忆体系：
1.  **引入 ChromaDB 或 Qdrant（轻量级向量数据库）** 到后端架构中。
2.  **存入记忆：** 每次对话结束后，后台利用 LLM 将核心事实（如“玩家今天过生日”、“吃了一顿火锅”）提取出来，向量化存入数据库。
3.  **读取记忆：** 每次你发送新消息前，先在向量库中进行相似度检索（RAG），查出过往相关记忆，并拼接进 `system_prompt` 中发给大模型。这样你的 AI 恋人就不会“失忆”了。
